# 问题分析方法论 Skill

系统化的问题根因分析方法论。当遇到 bug、异常行为、结果与预期不符、"为什么不工作"、需要调试定位根因，或发现自己陷入"反复假设-修改-失败"循环时，强制遵循"证据优先、追完整数据流、一次一变量、先查自己代码再怀疑外部"的纪律，杜绝盲猜。

---

## 概述

此 Skill 是一个**纪律清单**，强制在分析问题时严谨，避免"假设 → 改 → 失败"的盲猜循环。当你在调试、定位根因、或发现自己开始反复猜测时，**停下来**按这个流程走。

它来自一次真实教训：调试一个"CLI 路径 LLM 工具调用失败"的问题时，连续下了 6 个错误假设（system prompt / tools 数量 / thinking mode / tools 顺序 / "模型非官方" / 找台阶），全失败——因为一直在应用层日志（不完整）上打转，没去看真正发出去的 HTTP body。最后用一行临时打印抓到真证据，5 分钟定位：是 CLI 特有的 adapter 把 tools 参数丢了。这个 Skill 把那次教训固化成纪律。

### 适用场景

- bug、异常行为、结果与预期不符
- "为什么不工作"、"为什么 X 行为和 Y 不一样"
- 调试、定位根因
- 对比"工作的路径 vs 失败的路径"找差异
- **发现自己已经在反复假设时**（这是最该停下、改用这个 Skill 的信号）

## 核心纪律（必须遵守）

### 1. 证据分级：推断 ≠ 结论

- 下"根因是 X"之前，必须有**直接、完整**的证据支撑。
- 如果只是推断，明确说"我怀疑 X，但还没验证"，**绝不斩钉截铁地下结论**。
- 警惕"模型 / 服务 / 第三方"这类你没法直接验证的归因——放到**最后**怀疑。

### 2. 追完整数据流，不中间归因

- 从输入到**最终实际输出**，每一层都确认字段对得上。
- **先搞清楚日志 / 证据记录的是哪一层**——绝不能假设"日志 = 完整请求 / 完整数据"。
  日志常常是中间层（如应用层 event，不是真正发出去的 HTTP body / 实际执行输入），
  会缺关键字段。在不完整的证据上反复对比，永远找不到根因。
- 找不到根因时，先反思："我是不是在错误的层打转？真正的证据在哪一层？我抓到的
  是'准备的数据'还是'真正用出去的数据'？"

### 3. "只一个路径失败" = 该路径特有的代码问题

- 如果 A 路径工作、B 路径失败，**第一怀疑对象是 B 路径特有的代码**
  （adapter、包装层、该路径独有的配置 / 转换），**不是**共性问题（模型、共享函数、
  共享配置）。
- 共性的东西在 A 能工作，就基本排除了——别再怀疑它。

### 4. 一次只改一个变量

- 最小改动 → 验证 → 再下一步。
- **绝不**同时改多个（system prompt + 配置 + 代码 + 参数），否则一个变量生效 / 失效
  你都分不清，永远定位不了。

### 5. 怀疑自己的代码，先于怀疑外部

- 先把自己能控制的每一层代码确认对（输入到输出的每一跳），再怀疑模型 / 服务 / 第三方。
- 尤其"模型不调工具""服务返回怪格式"这种归因，几乎总是自己中间某层把数据弄丢了
  或弄错了，而不是模型 / 服务的问题。

### 6. 不给自己找台阶

- "日志没记这个字段""可能是模型的问题""这个不好验证"——这些话一出现，
  立即转成"为什么没记？怎么直接验证？我能在哪一层抓到真数据？"，
  而不是绕过去、换个假设继续猜。

## 标准流程

1. **抓最终实际输出**：先看真正发出去 / 执行的数据（HTTP body、实际传给下游的参数、
   实际落盘的结果），不是中间层日志。明确确认：我手里的证据是哪一层？
2. **对比工作 vs 失败**：找一条**确定工作**的路径，和失败的路径**逐层、逐字段**
   对比完整数据（包括日志没记的字段——用临时打印 / eprintln / 断点抓）。
3. **定位分歧点**：从哪一层开始两边不一样？分歧点 = 嫌疑根因。
4. **最小验证**：在分歧点做**最小、单一**改动，验证假设。
5. **确认或回退**：验证通过 = 根因确认，修复；不通过 = 回到第 1 步，反思是不是
   看错了层、抓错了证据。

## 反模式（必须避免）

- ❌ 反复"假设 → 改 → 失败"循环，每次都没充分验证就下斩钉截铁的结论
- ❌ 在错误的层打转（中间层日志 ≠ 真证据），却以为已经"看过了"
- ❌ 甩锅外部（模型 / system / 服务）先于查自己路径特有的代码
- ❌ 同时改多个变量
- ❌ 找台阶（"日志没记""可能模型问题"）代替直接验证
- ❌ 被用户 / 同事质疑时绕，而不是顺着质疑往下挖——质疑往往指向你跳过的那一层

## 下结论前的自检（5 问）

1. 这是"我**验证过**的"还是"我**推断**的"？
2. 我手里的证据是**哪一层**？是最终实际输出，还是中间层？
3. 工作路径 vs 失败路径，差异我**逐层**确认了吗（含日志没记的字段）？
4. 失败路径有没有**自己特有的代码**（adapter / 包装 / 转换）我还没查？
5. 我这次打算改**几个**变量？（>1 个就先减到一个）

## 典型教训（真实案例）

> **现象**：CLI 路径的 LLM 工具调用失败（模型用文本"表演"工具调用），gateway 路径正常。
>
> **走过的弯路**：反复假设 system prompt / tools 数量 / thinking mode / tools 顺序 /
> "模型非官方"，每次都改了再测，全失败——因为一直在**应用层日志**（它记的是 agent
> 准备的 tools=39）上对比，而真正的 HTTP body 里 tools=0。
>
> **真根因**：CLI 特有的 `ProviderAdapter.chat` 把 tools 参数丢了（参数名 `_tools`
> 丢弃 + 传 `&[]`）。gateway 用的是另一个 adapter，所以正常。**只 CLI 失败 = CLI
> 特有代码问题**，第一时间就该查 CLI 的 adapter，而不是怀疑模型。用一行 eprintln
> 抓真正的 HTTP body，5 分钟就定位了。

---

**一句话总结**：证据要追到"真正用出去的那一层"，差异要逐层对比，一次只改一个变量，
自己的代码先于外部被怀疑。
