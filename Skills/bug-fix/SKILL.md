# 改 BUG 纪律 Skill

> **改 bug / 修 bug / fix bug / 改既有功能行为**时强制遵循的工程纪律。
>
> 动代码前先**定真相源与不变量、读代码定位根因到具体位置、评估影响**；改时**维持单一真相源、清理对称彻底**；改完**横向扫同类一并修**。杜绝"补丁式修 bug → 拆东墙补西墙 → 改一个 bug 出一个新 bug"。
>
> **当你发现自己改了又坏、坏了又改，或要修任何 bug/改任何既有行为时，停下来按这个走。**

本 Skill（以及任何 Skill / 纪律同理）**只有登记进自动加载的上下文（`CLAUDE.md` 的 Skills 列表）才会被后续 session 发现和遵循**。内容写得再对，不登记就是磁盘上的死文件——下次依然找不到、依然会犯同样的错。

## 概述

此 Skill 是一个**纪律清单**，强制在改任何 bug / 做任何修改时严谨，避免"假设→改→出新 bug"的循环。

它和 [`problem-analysis`](../problem-analysis/SKILL.md) 互补：
- `problem-analysis` 侧重**怎么找到根因**（证据分级、追数据流、一次一变量）。
- 本 Skill（`bug-fix`）侧重**改的对、改得全、不引入新 bug**（真相源、不变量、横向扫、清理彻底）。

### 适用场景
- 修任何 bug（前端 / 后端 / 配置 / 数据）。
- 改既有功能行为。
- 发现自己改了又坏、坏 了又改的循环。
- 改一个地方前/后，评估有没有同类也该一起改。

## 心法（贯穿全程）

- **不猜**：有代码就能完全确认。怀疑某处有问题，直接读代码定位到具体位置再下结论；遇到无法确认的，提疑问、求助，**绝不凭推断动手**。
- **单一真相源**：任何概念（一个开关、一个默认值、一个状态）只能有一个权威定义，其余都是它的视图。**两套并存就是 bug 源**。
- **先根因后方案**：方案必须在"根因确认 + 影响明确 + 代码验证"三者齐备后才提。

## 标准流程

### 一、定位根因（动代码前必做）
1. **定真相源和不变量**——这个状态/行为谁是权威？什么必须始终成立？（例：默认模型以 `agents.defaults.llm` 为准，不是 `model_list[0]`；语音开关以后端 session 为准，不是 UI ref 或 config。）
2. **读代码定位，不靠推断**——症状在 A 层，证据要追到**真正用出去的那一层**（落盘的数据、发出的请求 body、实际执行的状态机），准确到代码位置。怀疑即读那处代码确认。
3. **理清数据流/状态流/生命周期**——数据谁产生、过哪些层、存哪、谁读写；状态何时创建/丢失/恢复（尤其 SPA 组件的 mount/unmount、后端 session 的 start/stop）。
4. **吃透原设计**——判断 bug 是"实现没跟上设计"（补齐实现即可）还是"设计本身有问题"（改设计），别上来另起一套。
5. **借鉴同类**——查项目里有没有类似功能已做过类似修改，能否复用其思路/模式（例：加引擎就照 STT/TTS 引擎那套：全局 state + load + apply + release）。

### 二、确认影响
6. 根因确认后，明确它影响**哪些路径/功能/字段**，不只盯被报告的那一个症状。

### 三、定方案
7. **维持单一真相源**——修法让真相源一致，不新增并存的"假真相"。
8. **清理对称彻底**——涉及生命周期/清理（unmount、stop、释放），所有相关项一起清，不留半拉子（UI 停了后端没停 = 撒谎）。
9. **提方案的前提**——可以给用户多个方案选，但前提是：根因已确认、影响已分析、已知信息已收集、且已在代码上验证。**没验证的方案不提。**

### 四、改完之后
10. **横向扫同类**——分析是否还有其他类似功能有**同样问题**，一并修。
11. **验证 + 如实汇报**——编译/测试通过再报；说清改了什么、为什么、还遗留什么/有什么疑问。

## 反模式（必须避免）

- ❌ **症状补丁**：治"卡死"就强停、治"橙灰"就 reset，没抓真相源/不变量 → 拆东墙补西墙，必出新 bug。
- ❌ **猜**：不读代码先列方案、凭推断下结论。
- ❌ **多套真相源并存**：位置 vs 字段、config vs session vs UI ref——一旦脱节就 bug。
- ❌ **清理半拉子**：unmount 停了 STT 漏 TTS；reset 了 UI 没停后端 → UI 撒谎。
- ❌ **孤岛修改**：改前没借鉴同类怎么做、改后没扫同类是否同病。

## 真实案例（反面教材）：语音按钮"拆东墙补西墙"

> 场景：dashboard chat 页有听写/对话/播放三个语音按钮。用户报告切页后按钮卡死。整个过程改了 3 轮，每轮治一个症状、出一个新 bug，直到抓住"真相源=后端 session、UI 必须如实反映它"这个不变量才一次性解决。

### 第 1 轮：卡死 → 强停（埋雷）
- **症状**：chat 开听写 → 切去别的页 → 切回来 → 听写按钮卡死、无法再选中。
- **真根因**：切走时后端 STT 会话**没停**（orphan）；切回来 ChatPanel 重挂载，UI reset=false；再点 start → 后端报 "already running" → 永远起不来。
- **错误修法**：在 `onUnmounted` 里强停 STT 会话（`stt_to_input_stop`）。
- **违反**：心法"先根因后方案"——没定不变量（UI 必须如实反映后端 session）就动手，只治了"orphan"这个症状。
- **后果**：治了卡死，但"切个页就把我的听写停了"，且只停 STT 没停 TTS → 埋下第 3 轮的雷。

### 第 2 轮：橙+灰 → reset（再埋雷）
- **症状**：播放按钮显示橙色（选中）+灰色（禁用）同时出现。
- **真根因**：按钮模板 `:class="{active: voicePlayback}"`（橙）和 `:disabled="!ttsReady}"`（灰）是**两个独立绑定**；initVoiceState 从 config 恢复 `voicePlayback=true` 但**不检查 ttsReady** → TTS 没就绪时橙+灰。
- **错误修法**：mount 时 `voicePlayback.value = false`（强制 reset）。
- **违反**：反模式"症状补丁"+"多套真相源"——voicePlayback（UI ref）、playback_enabled（config）、后端 session 三套状态；reset 只动了 UI ref 这一套。
- **后果**：治了橙灰，但破坏了"切回来还开着"，且 UI reset 成关、后端还在播 → UI 撒谎。

### 第 3 轮：UI 撒谎 → 终于抓不变量
- **症状**：切页回来按钮全关，但后端 TTS/STT 还开着（用户根本没动 TTS/STT 总开关）。
- **真根因**：前两轮叠加——reset 破坏持久、`onUnmounted` 只停 STT 漏 TTS。
- **正解（一次到位）**：
  1. 定真相源=后端 session；不变量=UI 必须如实反映它。
  2. 选一致的实现策略：**走的时候把三个会话全停**（STT 听写/对话 + TTS 播放都在 `onUnmounted` 里 stop），UI 按停后的真状态显示关。
  3. 对称彻底：补上漏掉的 `tts_playback_stop`。
- **要点**：第 1 轮就该这么想——"UI 必须如实反映后端"。当时若抓牢这条，要么"走时全停+UI 如实关"、要么"走时不停+UI 从后端查真值恢复"，都不会走到第 3 轮。

### 教训映射
| 轮次 | 违反的纪律 |
|---|---|
| 第 1 轮 | 心法"先根因后方案" + 流程一.1"定不变量" |
| 第 2 轮 | 反模式"症状补丁"+"多套真相源" |
| 第 3 轮 | 反模式"清理半拉子"（第 1 轮的 onUnmounted 没对称停 TTS） |

> 同一调试链里的另一个反面案例：TTS 遇 emoji 抛 "Unknown token"。先**猜**是"全角标点"（错，改了没用）；用户做 A/B 对照（有 😄 崩 / 没 😄 行）+ 看 `normalized_text` 日志，才锁定是 emoji 被剥后留下的"孤儿空格"让 Kokoro 中文 lexicon 抛异常。**教训**：别猜，做 A/B 拿直接证据，日志要打出"真正喂给下游的数据"。

## 下结论/动手前自检

1. 真相源和不变量定了吗？
2. 根因追到**代码位置**了吗（还是停在推断）？
3. 影响范围明确了吗？
4. 同类功能看过没（借鉴 / 横扫）？
5. 提的方案在代码上验证过了吗？

**一句话总结**：先定真相源和不变量，读代码定位根因到具体位置，评估影响，借鉴同类定方案，改完横扫同类一并修——治本不治标，别拆东墙补西墙。
