# 结构化开发流程 (Structured Development Process)

此 skill 定义了严格的软件开发流程，所有开发任务必须按照此流程执行，确保代码质量和项目稳定性。

## 📋 流程概述

完整的开发流程包含以下阶段：

1. **预研阶段** - 方案预研与分析
2. **评估阶段** - 等待用户评估预研方案
3. **计划阶段** - 制定详细开发计划
4. **开发阶段** - 按计划逐步开发
5. **测试阶段** - 单元测试、集成测试、系统测试
6. **整理阶段** - 整理开发流程和内容
7. **复查阶段** - 二次检测残留问题和新问题
8. **修复阶段** - 如有问题，修复并重新测试
9. **迭代阶段** - 继续检查直到无问题
10. **报告阶段** - 生成完整报告并汇报

---

## 📖 详细流程说明

### 阶段 1: 预研阶段（方案预研）

**目标**: 在动手开发前，充分理解问题并设计解决方案。

**执行步骤**:

1. **问题分析**
   - 明确需求是什么
   - 分析当前代码状态
   - 识别技术难点和风险

2. **方案设计**
   - 设计技术方案
   - 考虑多种实现方案
   - 评估各方案的优劣
   - 推荐最优方案

3. **影响评估**
   - 分析对现有代码的影响
   - 评估可能引入的风险
   - 制定风险应对策略

4. **生成预研文档**
   - 创建 `docs/PLAN/YYYY-MM-DD_FEATURE_NAME.md`
   - 包含：问题分析、方案设计、影响评估、推荐方案

**输出**: 预研方案文档

---

### 阶段 2: 评估阶段（等待用户评估）

**目标**: 获得用户对预研方案的批准。

**执行步骤**:

1. 向用户展示预研方案
2. 等待用户评估和反馈
3. 根据用户反馈调整方案（如需要）

**重要**: ⚠️ **在用户批准预研方案之前，不得进行任何开发工作！**

---

### 阶段 3: 计划阶段（制定开发计划）

**目标**: 将方案转化为可执行的开发计划。

**执行步骤**:

1. **任务分解**
   - 将功能分解为具体的任务
   - 确定任务依赖关系
   - 估算每个任务的工作量

2. **制定步骤**
   - 明确每个步骤的具体操作
   - 确定每个步骤的验收标准
   - 识别关键路径

3. **生成开发计划**
   - 创建 `docs/PLAN/YYYY-MM-DD_FEATURE_NAME_开发计划.md`
   - 包含：任务列表、步骤说明、验收标准、测试计划

**输出**: 开发计划文档

---

### 阶段 4: 开发阶段（按计划开发）

**目标**: 按照开发计划逐步实现功能。

**执行步骤**:

1. **严格按计划执行**
   - 每次只处理一个步骤
   - 完成当前步骤后再进入下一步
   - 记录每个步骤的执行情况

2. **代码质量**
   - 遵循项目代码规范
   - 保持代码简洁清晰
   - 添加必要的注释

3. **开发日志**
   - 记录每个步骤的修改
   - 记录遇到的问题和解决方案
   - 保存到开发计划文档中

**输出**: 完成的代码 + 开发日志

---

### 阶段 5: 测试阶段（多层次测试）

**目标**: 确保代码质量，及早发现问题。

**执行步骤**:

1. **单元测试**
   - 为每个新函数编写单元测试（`#[cfg(test)] mod tests` 内联）
   - 确保测试覆盖率达到要求
   - 运行：`cargo test -p <crate-name>`

2. **集成测试**
   - 测试模块间的交互
   - 验证接口正确性
   - 运行：`cargo test -p integration-test`

3. **系统测试**
   - 测试完整的功能流程
   - 验证非功能需求（性能、安全性等）
   - 进行端到端测试

4. **回归测试**
   - 运行所有现有测试
   - 确保没有破坏现有功能
   - 运行：`cargo test --workspace`

5. **编译验证** ⚠️ **新增**
   - **必须验证项目整体编译成功！**
   - 运行项目编译命令
   - 检查所有 crate 是否编译通过
   - 验证无编译错误、无警告
   - 运行：
     ```bash
     # Rust 项目整体编译
     cargo build --release -p nemesisbot

     # 或检查整个 workspace
     cargo check --workspace
     ```

**验收标准**:
- ✅ 所有测试通过
- ✅ 项目整体编译成功
- ✅ 无编译错误和警告

**输出**: 测试报告

---

### 阶段 6: 整理阶段（整理开发内容）

**目标**: 归档开发过程，方便后续查阅。

**执行步骤**:

1. **整理代码变更**
   - 列出所有修改的文件
   - 说明每个文件的变更内容

2. **整理开发过程**
   - 记录开发中的关键决策
   - 记录遇到的问题和解决方案
   - 记录重要经验教训

3. **更新文档**
   - 更新相关技术文档
   - 更新 API 文档（如需要）
   - 更新用户文档（如需要）

4. **保存整理文档**
   - 创建 `docs/REPORT/YYYY-MM-DD_FEATURE_NAME_开发整理.md`
   - 包含：代码变更清单、开发过程、文档更新

**输出**: 开发整理文档

---

### 阶段 7: 复查阶段（二次检测）

**目标**: 发现残留问题和引入的新问题。

**执行步骤**:

1. **残留问题检测**
   - 检查是否有未完成的功能
   - 检查是否有临时代码未清理
   - 检查是否有 TODO/FIXME 未处理

2. **新问题检测**
   - 代码审查：检查代码质量
   - 安全审查：检查安全漏洞
   - 性能审查：检查性能问题
   - 并发审查：检查并发安全问题

3. **生成复查报告**
   - 记录发现的问题
   - 评估问题严重程度
   - 创建问题清单

**输出**: 复查报告（包含问题清单）

---

### 阶段 8: 修复阶段（如有问题）

**目标**: 修复发现的问题。

**执行步骤**:

1. **问题修复**
   - 按优先级修复问题
   - 对每个问题进行根因分析
   - 实施修复方案

2. **回归测试**
   - 重新运行所有测试
   - 确保修复没有引入新问题
   - 验证问题确实被解决

**重要**: 修复后必须重新测试！

**输出**: 修复的代码 + 测试结果

---

### 阶段 9: 迭代阶段（继续检查）

**目标**: 持续改进，直到没有问题。

**执行步骤**:

1. **重新复查**
   - 回到阶段 7
   - 再次进行残留问题检测
   - 再次进行新问题检测

2. **判断**
   - 如果发现问题 → 回到阶段 8 修复
   - 如果没有问题 → 进入阶段 10

**重要**: 此循环持续直到没有问题为止！

---

### 阶段 10: 报告阶段（生成最终报告）

**目标**: 向用户汇报完整开发过程。

**执行步骤**:

1. **汇总所有信息**
   - 预研方案
   - 开发计划
   - 代码变更
   - 测试结果
   - 复查结果
   - 修复记录

2. **生成最终报告**
   - 创建 `docs/REPORT/YYYY-MM-DD_FEATURE_NAME_最终报告.md`
   - 包含：问题背景、解决方案、实施过程、测试结果、问题修复、最终状态

3. **向用户汇报**
   - 总结开发成果
   - 报告测试结果
   - 说明已知问题（如有）
   - 提出后续建议

**输出**: 最终报告 + 汇报给用户

### 文档命名规范

1. **头部信息**

   - 创建人
   - 版本号

2. **章节结构**
   - 清晰的章节划分

   - ✅ 已完成
   - ⏳ 进行中
   - ❌ 未完成
   - ⚠️ 需要注意

## 🧪 测试文件规范

### 测试目录结构

Rust 项目使用内联测试（`#[cfg(test)] mod tests`）和集成测试两种方式：

```
crates/
├── nemesis-cluster/                    # 集群模块
│   ├── src/
│   │   ├── cluster.rs                  # 内含 #[cfg(test)] mod tests
│   │   ├── continuation_store.rs       # 内含 #[cfg(test)] mod tests
│   │   └── rpc/
│   │       ├── client.rs              # 内含 #[cfg(test)] mod tests
│   │       └── server.rs             # 内含 #[cfg(test)] mod tests
│   └── tests/                          # 集成测试（如有）
├── nemesis-channels/                   # 通道模块
│   └── src/
│       └── manager.rs                  # 内含 #[cfg(test)] mod tests
└── nemesis-security/                   # 安全模块
    └── src/
        └── scanner.rs                  # 内含 #[cfg(test)] mod tests

test-tools/
├── integration-test/                   # CLI 集成测试（298 个断言）
├── cluster-test/                       # P2P 集群测试
├── e2e-tests/                          # 端到端 AI 管线测试
└── test-harness/                       # 测试辅助库
```

### 测试文件命名规范

| 测试类型 | 位置 | 说明 | 运行命令 |
|---------|------|------|---------|
| 单元测试 | `crates/<name>/src/*.rs` 内 `#[cfg(test)] mod tests` | 内联在源码中 | `cargo test -p <crate-name>` |
| 集成测试 | `crates/<name>/tests/*.rs` | 独立测试文件 | `cargo test -p <crate-name>` |
| CLI 集成测试 | `test-tools/integration-test/` | 端到端 CLI 测试 | `cargo test -p integration-test` |
| 集群测试 | `test-tools/cluster-test/` | P2P 集群测试 | `cargo test -p cluster-test` |

### 测试编写规范

1. **内联测试（推荐）**
   ```rust
   // 在 src/some_module.rs 文件底部

   #[cfg(test)]
   mod tests {
       use super::*;

       #[test]
       fn test_something() {
           let result = some_function();
           assert!(result.is_ok());
       }

       #[tokio::test]
       async fn test_async_function() {
           let result = async_function().await;
           assert!(result.is_ok());
       }
   }
   ```

2. **集成测试**
   ```rust
   // 在 crates/<name>/tests/integration_test.rs

   use nemesis_cluster::Cluster;

   #[tokio::test]
   async fn test_cluster_start() {
       let dir = tempfile::tempdir().unwrap();
       let cluster = Cluster::new(dir.path()).unwrap();
       cluster.start().await.unwrap();
       cluster.stop().await;
   }
   ```

3. **运行特定测试**
   ```bash
   # 运行单个测试函数
   cargo test -p nemesis-cluster test_cluster_start

   # 运行某个 crate 的所有测试
   cargo test -p nemesis-agent

   # 运行整个 workspace 的测试
   cargo test --workspace
   ```

## ⚠️ 重要原则

### 必须遵守的原则

1. **预研优先**
   - 不做预研，不做开发！
   - 预研方案不批准，不做开发！

2. **计划驱动**
   - 没有计划，不做开发！
   - 严格按计划执行，不跳步骤！

3. **测试为王**
   - 没有测试，不算完成！
   - 测试不通过，不算完成！

4. **质量第一**
   - 有问题必须修复！
   - 不带问题进入下一阶段！

5. **文档完整**
   - 没有文档，不算完成！
   - 文档与代码保持一致！

6. **编译验证** ⚠️ **新增**
   - **开发完成后必须保证项目整体编译成功！**
   - 修改代码后必须验证编译是否通过
   - 删除方法或修改 trait 时，必须检查所有调用方
   - 添加新方法时，必须更新依赖的调用代码
   - **编译失败就是开发未完成！**

**编译验证步骤**：
1. 运行项目编译命令
   ```bash
   # Rust 项目
   cargo build --release -p nemesisbot

   # 或快速检查
   cargo check --workspace
   ```
2. 检查编译输出
   - ✅ 无错误：继续下一步
   - ❌ 有错误：立即修复，重新编译
3. 修复所有编译错误
4. 重新验证编译直到成功

**重要**：只有编译成功，才能进入下一阶段！

### 禁止事项

❌ **禁止**跳过预研直接开发
❌ **禁止**预研方案未批准就开始开发
❌ **禁止**没有开发计划就开始编码
❌ **禁止**跳过测试或测试不通过就进入下一阶段
❌ **禁止**发现问题不修复就进入下一阶段
❌ **禁止**没有文档就认为任务完成
❌ **禁止**编译失败就认为开发完成** ⚠️ **新增**
❌ **禁止**删除方法后不检查调用方** ⚠️ **新增**
❌ **禁止**修改 trait 后不更新依赖代码** ⚠️ **新增**

## 🎯 使用指南

### 当用户要求开发新功能时

1. **第一步**: 告知用户将开始预研阶段
2. **第二步**: 进行问题分析和方案设计
3. **第三步**: 生成预研报告并提交用户评估
4. **第四步**: 等待用户批准
5. **第五步**: 批准后制定开发计划
6. **第六步**: 按计划逐步开发和测试
7. **第七步**: 复查和修复问题
8. **第八步**: 生成最终报告并汇报
